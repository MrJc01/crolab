### 1.3 Problemas Reais Encontrados

1. **`run` é inútil para o usuário final.** A imagem Docker está hardcoded como `alpine:latest` e o comando como `echo`. O usuário não escolhe nada — nem imagem, nem comando, nem máquina destino.

2. **`auth` e `billing` são fachada.** Não comunicam com nenhum backend. São `fmt.Printf`. Isso é aceitável como scaffold, mas precisa estar claro no roadmap.

3. **`StreamLogs` é fake.** O server retorna 4 strings hardcoded. Não há conexão real entre o Docker runner e o stream de logs. O job roda assíncrono e os logs nunca voltam pro cliente.

4. **JobID é frágil.** `fmt.Sprintf("job-%d", len(req.Payload))` — dois payloads do mesmo tamanho geram o mesmo ID. Colisão garantida.

5. **`runCmd` não aceita `--image` nem `--cmd`.** O usuário não tem como especificar o que executar.

6. **E2E test não testa Docker.** Ele usa o binary compilado mas o server mock responde logs fake — então o teste prova que o gRPC funciona, não que jobs executam.

7. **Sem failover real.** `GetServer` retorna o default e pronto. Se falhar, morre. O failover mencionado em docs não existe no código.

8. **Sem sort por prioridade.** `config ls` lista na ordem de inserção, não por prioridade.

9. **Indentação inconsistente no server.go** (linhas 32-34 usam espaços em vez de tabs).

## Parte 2 — O Que o Projeto Quer Ser (Visão Consolidada)

Baseado em toda a conversa, o Crolab quer ser:

```
┌─────────────────────────────────────────────────┐
│              CROLAB PLATFORM                     │
│                                                  │
│  Developer CLI ──► Orchestrator ──► GPU Nodes    │
│       │                │                         │
│       │          ┌─────┴──────┐                  │
│       │          │            │                   │
│     TUI        Crom Cloud   P2P Direct           │
│   Monitor      (SaaS)      (Self-hosted)         │
│                  │            │                    │
│              Billing      Hash Auth              │
│              Stripe       gRPC Token             │
│                  │            │                   │
│              ┌───┴────┐   ┌──┴───┐              │
│              │Vast.ai │   │Local │               │
│              │RunPod  │   │LAN   │               │
│              │AWS/GCP │   │VPN   │               │
│              └────────┘   └──────┘              │
└─────────────────────────────────────────────────┘
```

**Três modos de uso:**
1. **Self-hosted puro:** Dev roda `serve` na própria máquina ou VM alugada, conecta via hash.
2. **Crom Cloud:** Dev compra créditos, a Crom roteia para máquinas que ela subloca.
3. **Orquestrador agnóstico:** Dev conecta contas de terceiros (Colab, Vast) e o Crolab gerencia.

## Parte 3 — Roadmap Real de Engenharia (Por Prioridade)

### Fase 2A: Tornar o `run` Utilizável (URGENTE)
O comando mais importante do produto não funciona para uso real.

- [ ] **Flags `--image` e `--cmd` no `runCmd`**
  - `crolab run . --image python:3.11 --cmd "python train.py"`
  - Se omitidos, detectar automaticamente (Dockerfile presente? usar `docker build`)
- [ ] **Seleção interativa de máquina no `run`**
  - Flag `--target <nome>` para escolher máquina específica
  - Sem flag: usa o default por prioridade
- [ ] **JobID com UUID real**
  - `github.com/google/uuid` ou `crypto/rand` hex
- [ ] **Logs reais do Docker conectados ao StreamLogs**
  - O `docker_runner.go` precisa capturar stdout/stderr e publicar num channel
  - O `server.go` StreamLogs precisa ler desse channel em vez de retornar strings fake

### Fase 2B: Config Robusto e Failover
- [ ] **`GetBestServer()` com sort por prioridade**
  - Retorna a lista ordenada por `Priority` (menor = melhor)
- [ ] **Failover real no `runCmd`**
  - Tenta servidor 1 → timeout/erro → tenta servidor 2 → loga a transição
- [ ] **`config ls` ordenado por prioridade**
  - Marcar o default com `*`
- [ ] **`config set-default <nome>`**
  - Comando para trocar o server padrão sem re-adicionar
- [ ] **Fix: indentação tabs no server.go**

### Fase 2C: TUI Monitor Mode
- [ ] **`crolab monitor`** — Dashboard interativo no terminal
  - Biblioteca: `github.com/charmbracelet/bubbletea`
  - Painel: Lista de nodes, status de conexão, jobs ativos
  - Seleção por setas, ações com Enter/Delete
  - Split-screen: metade superior = nodes, metade inferior = logs

### Fase 3A: Auth Real (Backend necessário)
- [ ] **Definir o backend Crom Cloud**
  - REST API mínima: `/register`, `/login`, `/billing/status`
  - Pode ser Go (gin/echo) ou Node.js — decisão do dono
- [ ] **CLI `auth login` real**
  - POST para o backend, recebe JWT, salva em Viper
- [ ] **CLI `billing` real**
  - GET saldo, POST compra de créditos

### Fase 3B: Hijack Script
- [ ] **`scripts/install.sh`**
  - `curl -sSL crolab.crom.run/install | bash`
  - Detecta OS, baixa binário, roda `crolab serve --generate-auth`
  - Printa o hash + IP para o dev copiar

### Fase 3C: Multi-Tenancy Real
- [ ] **Configuração de slots no `serve`**
  - `crolab serve --slots 2 --cpus-per-slot 4 --mem-per-slot 8g`
  - O semáforo atual (hardcoded 2 slots) vira configurável
- [ ] **GPU slicing via NVIDIA MPS ou `CUDA_VISIBLE_DEVICES`**
  - Para fracionar GPU real, não apenas CPU/RAM

### Fase 4: Frontend Web
- [ ] **Dashboard web (Next.js ou Nuxt)**
  - Espelha as funcionalidades do CLI
  - Gerenciamento visual de nodes
  - Notebooks estilo Colab (médio prazo)
